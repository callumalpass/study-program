## Topic Map

Design patterns are names for recurring design moves, not requirements to add boilerplate. This topic emphasizes Factory, Observer, Strategy, Decorator, and Adapter as tools for reducing coupling in specific situations.

## Learning Objectives

- Explain the problem a pattern solves before implementing it
- Use Factory, Observer, Strategy, Decorator, or Adapter when they simplify change
- Recognize Singleton as a cautionary pattern because it often hides global mutable state
- Prefer a simple function, explicit dependency, or plain class when a named pattern adds no value
- Identify anti-patterns such as God Object and unnecessary service locators

## Design Focus

The best pattern use is restrained. A design is better when it is easier to test, change, and explain, not when it contains more pattern names.
